# 个人任务与账单智能助理系统需求说明

## 1. 项目目标

开发一套可运行于 Windows 和 Linux 环境的个人智能助理系统，包含：

1. **服务端**
   - 负责任务数据、任务状态跟踪、账单数据、账单聚合结果的统一管理。
   - 底层使用 **SQLite** 作为基础数据库。
   - 提供可供桌面客户端和 Agent 框架调用的标准接口。
   - 部署在 Linux 服务器。

2. **桌面客户端**
   - 运行于 Windows 和 Linux。
   - 连接远程服务端，查看与管理任务、查看财务信息。
   - 在桌面上提供多个样式的日历控件，支持透明度调整、任务标记、快捷弹窗查看和新增任务。

3. **智能处理能力**
   - 通过 **LiteLLM** 接入大模型能力，用于账单合并、重复交易识别、同类交易归并、消费分类等。

4. **Agent 集成能力**
   - 服务端的任务接口与财务查看接口，需要可被 OpenClaw 等 Agent 框架通过独立 skill 调用。
   - skill 需支持配置服务端地址。

5. **安全要求**
   - 服务端仅允许通过认证后的 SSH Key 或 HTTPS 证书校验方式接入，避免外部未授权访问。

## 2. 系统范围

本系统分为四个核心子系统：

1. 任务管理服务
2. 账单采集与归并服务
3. 桌面日历与任务客户端
4. Agent/Skill 接入层

第一阶段范围外但应预留扩展能力：手机端 App、多用户协作、银行直连接口、自动化报税或财务报表导出。

## 3. 总体架构要求

### 3.1 架构形态

系统采用客户端-服务端架构：

- 服务端：部署于 Linux 服务器，提供 API、账单处理、数据存储、认证与审计。
- 桌面客户端：部署于 Windows / Linux，连接服务端获取任务和账单数据。
- 大模型服务层：通过 LiteLLM 调度外部模型服务，完成账单归并和分类。

### 3.2 数据存储

基础数据库为 SQLite，至少存储以下数据：

- 待办事项
- 任务子项
- 任务状态变更记录
- 日历关联信息
- 原始账单导入记录
- 标准化后的交易记录
- 合并后的账单记录
- 消费分类结果
- 用户配置（服务端地址、认证信息、界面偏好）
- 操作日志与审计日志

### 3.3 部署方式

- 服务端必须支持 Linux 部署。
- 客户端必须支持 Windows 和 Linux。
- 客户端需支持配置服务端地址。
- 系统应预留本地开发、测试、生产三种环境配置方式。

## 4. 用户角色

第一阶段默认单用户，但角色逻辑需预留扩展：

1. **系统主人/主用户**
   - 管理任务
   - 查看日历
   - 导入与查看账单
   - 触发账单归并与分析
   - 配置服务地址与认证方式

2. **Agent 调用方**
   - 通过 skill 调用任务接口
   - 查询任务、更新任务、添加任务
   - 查询财务汇总结果
   - 触发账单分析流程（按权限控制）

## 5. 任务管理需求

### 5.1 任务基础信息

每个任务至少包含以下字段：任务 ID、标题、详细描述、所属日期/开始时间/截止时间、当前状态、优先级、标签、创建时间、更新时间、是否置顶、备注。

### 5.2 任务状态

任务状态至少支持：预约、处理中、挂起、完成、取消。

要求：支持状态切换、记录状态变更历史、查看完整状态轨迹。

### 5.3 子项管理

每个任务可包含多个子项，子项至少支持：标题、描述、状态、排序、完成标记。

要求：支持新增、编辑、删除、重排，支持父任务状态与子项完成情况联动，支持快速勾选完成。

### 5.4 日历关联

任务必须可与某一天或某个时间段关联。

要求：某日期存在任务时，日历对应日期必须有特殊标记，需区分未完成、进行中、已完成任务。点击某一日期后，应弹出窗口显示当日任务列表、状态、摘要和可执行操作，并支持从日期弹窗快速新增任务。

### 5.5 任务操作

客户端和接口至少支持：新建任务、编辑任务、删除任务、修改状态、新增/编辑/删除子项、按日期查询任务、按状态查询任务、按关键字搜索任务、查看任务历史记录。

## 6. 桌面客户端需求

### 6.1 平台要求

客户端需同时支持 Windows 和 Linux。

### 6.2 桌面日历控件

至少包括：月视图、周视图（建议）、紧凑悬浮视图、透明背景小组件样式。

要求：支持切换样式、调整透明度、固定桌面位置、窗口置顶或嵌入桌面模式、浅色/深色主题、字号颜色间距配置。

### 6.3 日历交互

要求：日期上有任务时显示特殊标记；点击日期后弹出详情窗口，展示任务标题、状态、时间、子项摘要、备注，并支持新增任务、编辑任务、修改状态、打开完整详情。

### 6.4 任务查看与编辑界面

客户端需提供完整任务管理界面，用于查看今日任务、本周/本月任务、按状态筛选、查看任务详情、编辑任务及子项、查看历史状态变更。

### 6.5 服务端连接配置

客户端必须支持配置：服务端地址、端口、协议类型、证书/Key 路径、连接测试。

要求：首次启动时引导配置，支持保存多个服务器配置（第一阶段可只启用一个），支持连接失败提示和重试。

## 7. 账单系统需求

### 7.1 目标

系统需要从指定服务器侧拉取账单数据，统一处理来自多个渠道的消费记录，并生成整理后的花销信息。

### 7.2 数据来源

至少支持：微信、支付宝、京东、拼多多，以及多个信用卡、多个借记卡导出的交易记录。

要求：支持不同来源的导出文件导入，支持服务器端统一拉取和处理，支持后续扩展新的数据源。

### 7.3 原始账单处理

系统应支持导入：交易时间、交易金额、收支方向、交易对手/商户名称、渠道来源、订单号/流水号、备注、卡号尾号、原始导入文件信息。

要求：原始数据必须保留，不可被覆盖；标准化处理结果需与原始记录建立映射关系；导入过程要记录日志和错误原因。

### 7.4 账单合并与去重

系统需支持重复交易识别、同类交易合并、分渠道对账。

要求：自动合并后必须保留原始明细；用户应能手动撤销某次合并结果；所有合并动作需有日志记录。

### 7.5 消费分类

系统应支持通过 LiteLLM 接入大模型，对交易进行消费分类。

要求：支持自动分类、手动修正分类、记录模型建议分类/最终确认分类/分类置信度，并支持分类规则和提示词配置。

### 7.6 财务结果展示

系统至少要能输出：月度总支出、月度总收入、分类支出统计、渠道支出统计、卡片/账户支出统计、重复/异常交易列表、大额交易提醒。

要求：支持按月查看账单汇总、查看某月详细交易清单，并支持按来源、分类、卡片、金额区间筛选。

## 8. 大模型与 LiteLLM 集成需求

### 8.1 使用场景

LiteLLM 集成至少用于：交易去重判断、相似交易归并、消费分类、异常交易辅助识别、商户名称归一化。

### 8.2 模型调用要求

模型服务地址、模型名称、Prompt 模板需可配置；请求与响应需记录审计日志（敏感信息按需脱敏）；支持失败重试与超时控制；支持模型不可用时回退到基础规则模式。

### 8.3 结果可解释性

模型处理结果至少应包含：建议标签或分类、合并/去重判断结果、简要理由、置信度或可信等级。

## 9. 服务端 API 需求

服务端需提供标准 API，供桌面客户端和 Agent Skill 调用。接口类别至少包括：任务接口、子项接口、状态历史接口、账单导入接口、账单查询接口、财务汇总接口、智能分析触发接口、配置与健康检查接口。

## 10. Agent Skill 接入需求

服务端任务接口和财务查看接口需可被 OpenClaw 等 Agent 框架调用。

至少拆分为两个独立 skill：

1. 任务管理 skill
2. 财务查看 / 分析 skill

每个 skill 至少支持：服务端地址、端口、认证方式、证书 / Key 路径、请求超时配置。

## 11. 安全需求

系统需要避免外部未授权访问。

要求：

- 服务端不应无保护地直接暴露给公网。
- 客户端访问服务端必须通过受控认证机制。
- 至少支持基于 SSH Key 的受信接入，或基于 HTTPS 双向证书/服务端证书校验的安全接入。
- 客户端必须校验服务端身份。
- 服务端推荐校验客户端身份。
- 配置错误或证书无效时应阻止连接。
- 禁止提供“忽略证书错误”作为默认选项。
- 敏感数据传输必须加密，日志不得明文泄露高敏感内容。

## 12. 非功能需求

- 客户端应支持断线重连。
- 服务端异常时客户端要有明确错误提示。
- 数据导入与分析任务应支持进度展示或状态查询。
- 月历视图与普通任务查询应具备良好响应速度。
- 大模型调用应支持异步处理。
- 账单数据源、任务状态、消费分类应可扩展。
- 后续应可替换 SQLite 为更强数据库而不大改业务层。

## 14. 第一阶段最小可交付范围（MVP）

1. Linux 服务端部署能力
2. SQLite 数据库与基础表结构
3. 基础任务 CRUD 和状态管理
4. 桌面月历控件与日期任务标记
5. 日期点击弹窗查看和新增任务
6. 客户端连接服务端地址配置
7. 基础账单导入能力（先支持文件导入）
8. 微信/支付宝/信用卡基础交易标准化
9. 基于规则 + LiteLLM 的初版分类与去重
10. 月度支出汇总查看
11. 基础认证与证书校验
12. 两个可供 Agent 调用的独立 skill

## 15. 第二阶段建议增强项

- 周视图 / 多视图桌面小组件
- 更强的消费分析报表
- 自动账单拉取调度
- 预算管理与超支提醒
- 任务提醒与通知中心
- 多账户、多身份支持
- 更完整的对账工作流
- 移动端同步

## 16. 关键待确认问题

1. 桌面客户端更偏向原生桌面应用还是跨平台 UI 框架实现。
2. 服务端 API 采用 REST、gRPC，或混合模式。
3. 安全接入优先方案采用 SSH 隧道 + 本地访问，还是 HTTPS/mTLS 直连。
4. 账单导入初期优先支持哪些具体文件格式。
5. 大模型供应商、成本上限和隐私边界。
6. 任务是否需要提醒、重复周期、依赖关系。
7. 财务分类是否需要用户自定义分类树。

## 17. 验收标准

1. Linux 服务端可成功部署并启动。
2. Windows 与 Linux 客户端可连接服务端。
3. 桌面日历可显示任务标记，并可点击查看详情。
4. 可创建、编辑、更新状态、删除任务，并支持子项。
5. 任务状态历史可查询。
6. 至少支持部分账单来源导入并保留原始记录。
7. 系统可完成交易标准化、基础去重、基础分类。
8. 可查看月度支出统计与明细。
9. Agent skill 可访问任务接口与财务接口。
10. 未通过认证的客户端无法访问服务端。
